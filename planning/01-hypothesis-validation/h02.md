# H2 — Stale scores as a driver of under-performing offers
**Group:** Problem existence (H1–H4) · **Tests assumption:** A1 (stale-data root cause, §1.2) · **Sub-problem:** P1 (stale subset)

## Hypothesis statement (source §5)
A material share (>X%, threshold set with Risk) of "under-performing" offers stem from stale scores rather than model error.

## Falsification condition (verbatim)
> "Migration is uncorrelated with score age"

## Validation method (source, verbatim)
Vintage analysis: time-since-last-score vs. subsequent segment migration.

## Required data
| Item | Use |
|---|---|
| M1 Decision engine outputs (with timestamps, model version) | Score dates → time-since-last-score per customer |
| M5 Segment migration history | Subsequent migration events (direction, size, timing) |
| U6 Bureau data snapshots and refresh dates | Distinguish internal score staleness from bureau-input staleness (source: "Stale-data analysis (H2)") |

## Analysis protocol (business/analytical terms)
1. **Threshold pre-registration:** the "X%" materiality threshold is set with Risk before the analysis runs (OQ-12) — charter §3 anti-goal-seeking rule.
2. **Cohort definition:** customers grouped by time-since-last-score bands at a reference date (bands agreed with Risk), within product and segment strata.
3. **Comparison logic:** upward-migration rate at next re-scoring by score-age band. If upward migration rises materially with score age, staleness is producing temporary under-classification the bank's own model would correct if invoked earlier (this is O3's mechanism). Cross-check against U6 to see whether staleness sits in the score or in its inputs.
4. **Interpretation guard:** migration correlated with score age supports H2; migration uncorrelated with score age falsifies it. Distinguish genuine profile improvement from cut-off changes during the window (use M2 via the H1 workstream to exclude regime-change periods).
